0.요약 (Executive Summary)

종합 평가: B+ (잘 설계된 정적 데이터 서비스, 성장 단계 진입)
데이터 출처 투명성·정직한 결측치 처리·자체 레이팅 분리라는 설계 철학이 코드 전반에 일관되게 구현되어 있고, 일일 자동 갱신 파이프라인까지 완성된 상태입니다. 반면 테스트가 생성된 데이터 검증에만 의존하고(파서·모델 단위 테스트 부재), 단일 5.9MB JSON 로딩, 1,300줄 단일 App 컴포넌트, Baduk-R 자체 이력 미보존(추세 그래프가 GoRatings 이력을 표시) 등 다음 단계 성장을 막는 구조적 부채가 명확합니다.

개선은 ① 안정성(파서 테스트·장애 내성) → ② 성능/구조(데이터 분할·컴포넌트 분리) → ③ 제품 기능(이력 보존·딥링크·SEO) → ④ 모델 고도화(Glicko/WHR·예측 보정) 순서를 권장합니다.

1.프로젝트 현황 개요

Baduk-R은 한국·중국·일본·대만 프로 바둑 기사의 레이팅, 승부 예측, 대국 일정, 뉴스/칼럼, 선수 프로필을 5개 언어(en/ko/ja/zh-Hans/zh-Hant)로 제공하는 완전 정적 웹 서비스입니다. 런타임에는 외부 API를 일절 호출하지 않으며, 모든 데이터는 빌드 전 scripts/update-data.mjs가 수집·가공해 JSON으로 내보냅니다.

2.2 강점과 약점

✅ 강점

  • 일관된 데이터 윤리 — 출처·약관·신뢰도·결측 사유를 전부 데이터에 남기는 설계가 코드와 테스트로 강제됨. 이 프로젝트의 핵심 차별점.
  • 자체 레이팅 독립성 — Baduk-R이 GoRatings 복제가 아님을 테스트로 검증(상위 50명 평균 차이 > 300).
  • 장애 내성 있는 파이프라인 — 소스별 try/catch + 상태 기록 + 이전 스냅샷 뉴스 폴백으로 일일 배포가 끊기지 않음.
  • 보안 설계 — API 키가 빌드 타임에만 사용되고 프런트 번들에 포함되지 않음.
  • 일정 중요도/뉴스 큐레이션의 결정론적 점수화 — 사유(reasons)까지 노출해 설명 가능.
  • 의존성 최소주의 — 런타임 의존성 3개(react, react-dom, lucide). 공급망 리스크 낮음.

⚠️ 약점 · 리스크

  • GoRatings 단일 장애점 — 선수 명단·게임 기록·이력의 원천이 GoRatings 정규식 파싱 하나. 메인 페이지 마크업이 바뀌면 main()이 try/catch 없이 실패하고 일일 갱신 전체가 중단됨.
  • 파서 단위 테스트 부재 — HTML fixture 기반 테스트가 없어 소스 마크업 변경을 사전에 감지 불가. 깨진 후 source_status로만 알게 됨.
  • 추세 그래프 불일치 — 랭킹·예측은 Baduk-R인데 테이블/프로필의 트렌드 차트(player.history)는 GoRatings 이력. 사용자에게 혼동을 줄 수 있는 정합성 문제.
  • Baduk-R 이력 미보존 — 매 배포가 스냅샷을 덮어써서 자체 레이팅의 시계열을 만들 수 없음(델타는 모델 재실행으로 근사 중).
  • 커버리지 편중 — 게임 그래프가 상위 117명 중심이라 중하위권 레이팅은 신뢰 불가(불확실도로 표시는 됨).
  • 이름 매칭 휴리스틱 — 정규화 동명이인 충돌 가능성. unresolved 보고는 되지만 오매칭 검출 장치는 없음.
  • 대만 데이터 공백 — 일정·뉴스 모두 비어 있어 4개국 서비스라는 포지셔닝 대비 약함.

3.개선 진행 방향 (로드맵)

각 단계는 독립 배포 가능하며, 앞 단계가 뒤 단계의 안전망이 되도록 순서를 잡았습니다.

Phase 1 안정성 · 회귀 방지 기반 (1~2주)

  • 파서 단위 테스트 도입 — 실제 응답을 fixture로 저장하고 Vitest로 parseRatingRows, parseKbaSchedule, parseNihonSchedule, parseChineseDateRanges, normalizeName 등을 검증. 마크업 변경을 CI에서 즉시 감지.
  • GoRatings 장애 시 이전 스냅샷 폴백 — 뉴스에만 있는 폴백 패턴을 선수/레이팅 수집까지 확장해 단일 장애점 제거. 폴백 사용 시 source_status에 stale 표기.
  • update-data.mjs 모듈 분할 — scripts/sources/(소스별 fetch+parse), scripts/model/(Baduk-R), scripts/translate/로 분리. 테스트 가능성과 소스 추가 비용이 동시에 개선됨.
  • CI에 pytest 추가 — README가 약속하는 품질 게이트와 실제 CI를 일치시킴 (또는 Python 테스트를 Node로 통합해 이중 관리 제거).

Phase 2 성능 · 프런트 구조 (1~2주)

  • 데이터 분할 로딩 — baduk-data.json을 ① 랭킹 코어(선수+비교, 즉시 로드) ② 선수별 상세(players/{id}.json, 선택 시 lazy fetch) ③ 일정/뉴스로 분리. pretty-print 제거(minify)만으로도 약 30% 절감, 분할 시 첫 로드를 수백 KB 수준으로 단축 가능.
  • App.tsx 컴포넌트 분리 — RankingTable / PredictorPanel / ProfilePanel / SchedulePanel / NewsPanel / SourceHub 단위로 분할하고 데이터 가공 로직을 훅으로 추출. CSS도 컴포넌트별로 분리.
  • UI 상태의 URL 동기화 + 영속화 — 언어를 localStorage(최초엔 브라우저 언어 감지)에 저장하고, 국가/모드/정렬/선택 선수를 해시 파라미터로 반영해 공유 가능한 링크 제공.
  • 테이블 UX — 컬럼 헤더 클릭 정렬, 220행 제한 대신 가상 스크롤 또는 "더 보기".

Phase 3 제품 완성도 (2~4주)

  • Baduk-R 이력 아카이빙 — 일일 갱신 시 own_latest.json 요약본을 날짜별로 누적 저장(저장소 또는 별도 브랜치). 이것으로 추세 그래프를 GoRatings 이력 대신 진짜 Baduk-R 시계열로 교체 — 현재 가장 큰 정합성 문제 해소.
  • 선수 딥링크 + SEO — #/player/{id} 라우트, OG/Twitter 메타 태그, hreflang, sitemap. 빌드 타임에 상위 선수 정적 페이지 프리렌더링(데이터가 이미 빌드 타임에 존재하므로 비용 낮음).
  • 번역 커버리지 확대 — CI의 번역 한도(뉴스 6 · 일정 0)를 상향하되, 이미 번역된 항목은 이전 스냅샷에서 재사용하는 캐시를 도입해 API 비용·시간을 통제.
  • 대만 소스 보강 — 대만기원(台灣棋院) 일정/랭킹 소스 조사·추가로 4개국 커버리지 완성.
  • 모바일 최적화 — 현재 hide-tablet/hide-mobile로 컬럼을 숨기는 방식에서, 모바일 전용 카드 레이아웃으로 전환.

Phase 4 모델 고도화 (지속)

  • 게임 그래프 커버리지 확대 — 상세 수집 대상을 117명에서 단계적으로 확대(또는 전체 선수의 게임 목록 페이지 수집)해 중하위권 레이팅 신뢰도 개선. 수집량 증가는 동시성·캐싱으로 흡수.
  • 모델 업그레이드 — Elo → Glicko-2 또는 WHR(시간 감쇠 내장)로 전환, 흑/백 승률 이점 추정 추가. METHODOLOGY.md에 명시된 "정직하게 추정 가능해질 때 공개" 원칙과 부합.
  • 예측 보정(calibration) 공개 — 과거 예측 vs 실제 결과의 Brier score/신뢰도 곡선을 자동 산출해 방법론 페이지에 게시. 프로젝트의 투명성 철학을 모델 성능까지 확장.
